# ADR-0002: Database Engine â€” MySQL 8.x â†’ PostgreSQL

## Status
Accepted.

## Context
`docs/business/RAAD_Phase3.2_Database_Design_v1.md` (Â§1, line 5) and `.claude/rules/database.md` #1
originally named **MySQL 8.x** as the approved database engine, with Alembic migrations and the
`asyncmy` async driver. As of this decision, the project migrates to **PostgreSQL** before further
backend development continues, so that the five still-scaffold-only modules (`transport_ops`,
`billing`, `notifications`, `platform_audit`, `reporting`, `video`) are built Postgres-native from
the start rather than migrated later.

A repository-wide migration assessment (conducted prior to this ADR) found real MySQL-specific
surface in exactly four already-implemented modules â€” `core/db`, `iam`, `organization`,
`fleet_device`, `tracking` â€” and identified two points where the existing schema carried a
MySQL-specific *workaround* rather than a portable pattern:

1. **`device_assignments.active_device_key` / `active_vehicle_key`** (`fleet_device`, Database
   Design Â§9 / line 222): two `Computed` (generated) columns implementing MySQL's documented idiom
   for emulating a partial-unique constraint, each backed by a unique index.
2. **Enum columns** (`user_role`, `user_status`, `org_type`, `billing_model`,
   `organization_status`, `region_status`, `vehicle_status`, `device_lifecycle_state`,
   `camera_position`, `geofence_event_type`): SQLAlchemy's generic `Enum`, which MySQL renders as
   an inline column-level `ENUM(...)`.

No production data exists yet (project is greenfield; none of the four existing Alembic revisions
has been applied against a live database outside development). This removes data-migration risk
from the decision entirely â€” the choice is schema-design-only.

## Decision
1. **Engine changes to PostgreSQL.** `.claude/rules/database.md` #1 and
   `RAAD_Phase3.2_Database_Design_v1.md` Â§1 are updated accordingly. Driver changes from `asyncmy`
   to `asyncpg`.
2. **Partial unique indexes replace the generated-column workaround.** `device_assignments`'
   one-active-binding-per-device/vehicle invariant is expressed with PostgreSQL's native partial
   unique index (`CREATE UNIQUE INDEX ... WHERE unassigned_at IS NULL`) instead of the two
   `Computed` columns. This is strictly the mechanism PostgreSQL provides for exactly this
   constraint shape â€” MySQL had no equivalent, which is why the generated-column emulation existed
   in the first place. The two `active_device_key`/`active_vehicle_key` columns are removed from
   the schema; the ORM model reflects `unassigned_at IS NULL` directly with no denormalized key
   column standing in for it.
3. **PostgreSQL native `ENUM` types are kept**, not replaced with `VARCHAR` + `CHECK`. This
   preserves the existing invariant (illegal values rejected at the column-type level) with no
   weakening of validation. The accepted tradeoff (documented here so it isn't rediscovered as a
   surprise later): adding a new enum value in PostgreSQL requires `ALTER TYPE ... ADD VALUE` via
   an explicit Alembic migration (`op.execute(...)`, not autogenerated) rather than MySQL's
   in-place inline `ALTER TABLE ... MODIFY COLUMN`. Every future enum extension (e.g. a new
   `device_lifecycle_state`) must include this manual step â€” flagged here so module authors don't
   assume `alembic revision --autogenerate` will catch it, since it won't.
4. **General principle for this migration:** MySQL-specific implementation patterns are not carried
   over where PostgreSQL provides a native, better-suited feature for the same requirement. Decision
   3 is the explicit exception â€” kept because it is already the *better* pattern (a real DB-level
   enum vs. an unconstrained string), not a MySQL workaround.
5. **`vehicle_positions`' composite primary key `(id, event_time)`** is retained. It exists because
   the partition key must be part of every unique key on a partitioned table â€” a rule PostgreSQL's
   declarative partitioning shares with MySQL's `RANGE` partitioning (verify against the specific
   PostgreSQL version in use before the actual `PARTITION BY RANGE` DDL is written in a later
   phase; not yet implemented on either engine). This is retained as a shared relational-partitioning
   constraint, not a MySQL-only artifact, so it is not in scope for removal under Decision 4.
6. ULID-as-`CHAR(26)` primary keys are unaffected by this ADR. `CHAR(26)` is ANSI-standard and
   already an independently-approved decision (Database Design Â§1, resolving Backend LLD Â§20.2) â€”
   this migration does not reopen it, and does not adopt PostgreSQL's native `UUID` type as a
   substitute.

## Consequences
- `.claude/rules/database.md` #1 and `RAAD_Phase3.2_Database_Design_v1.md` Â§1 are updated in the
  same change that lands this ADR.
- `RAAD_Phase3.2_Database_Design_v1.md` Â§9 (`device_assignments`) is updated to describe the
  partial unique index instead of the generated-column pair.
- `pyproject.toml` drops `asyncmy`, adds `asyncpg`.
- `core/db/mixins.py` and the `iam`/`fleet_device`/`tracking` `infra/models.py` modules drop
  `sqlalchemy.dialects.mysql` imports (`DATETIME`, `DECIMAL`) in favor of portable SQLAlchemy
  column types.
- The four existing Alembic revisions (iam, organization, fleet_device, tracking) are regenerated
  against PostgreSQL rather than hand-ported, since none has been applied to a real database yet â€”
  regeneration carries materially less risk than manual MySQLâ†’Postgres DDL translation.
- The `fleet_device` ORM model and its migration lose the two `Computed` columns; a new partial
  unique `Index(..., postgresql_where=...)` is added in their place.
- Scale path for `vehicle_positions` (Database Design line 478, "migrate to TimescaleDB") becomes
  strictly easier post-migration, since TimescaleDB is a PostgreSQL extension rather than a
  separate engine â€” noted as a positive side effect, not a driver of this decision.
- Every module still scaffold-only at the time of this ADR (`transport_ops`, `billing`,
  `notifications`, `platform_audit`, `reporting`, `video`) is built against PostgreSQL from its
  first migration â€” no retrofitting needed for those.
- `docker/docker-compose*.yml`'s commented-out `mysql:` service stub is updated to `postgres:16`
  when service definitions are eventually filled in (currently still a placeholder either way).
